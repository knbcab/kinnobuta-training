# KINNOBUTA Group Staff Training AI System
## Design Document v0.1

> 作りながら設計は変わる。これは「Claude Codeに渡したときに迷わずスタートできる粒度」を目指した、**変わることを前提とした**設計図。

---

## 1. このシステムが解いている問題

### 解いている問題

KINNOBUTAグループ（KINNOBUTA / ZETTON / HIBI COFFEE、3ブランド・スタッフ31名以上）において、新人スタッフを「KINNOBUTAグループのスタッフとして自走できる状態」にするまでの時間とエネルギーが、店長の口頭教育に依存している。

その結果、

- 店長によって伝わり方が違う（人フィルター問題）
- 店長の時間が消費される
- 新人は「困ったら聞く」しかできず、何が分からないかも分からない
- HIBI standard（自分たちの基準）が、教える人を経由するうちに薄まる

### 解こうとしているもの

新人が独力でグループの思考OSをインストールできる装置をつくる。店長フィルターを外し、Michaelさんが直接設計した思考の道筋を全員が同じように通る。

### 解いていない問題（重要）

このシステムは以下を **解こうとしていない**：

- やる気の見極め（Day 0物理スクリーニングが既に解いている）
- 店舗OJT（座学卒業後、人間が引き継ぐ）
- 採用判断（合否は最終的に人間 = Michael + 店長）

これらをAIに任せようとすると設計が崩れる。AIは**思考のセットアップ**だけに集中する。

## 2. 全体構造：3層スクリーニング

```

                       ↓
┌─────────────────────────────────────────────────────┐

│  - スタッフが自分のPC/タブレットで一人で進める          │
│  - 各問にAI解説（KINNOBUTA文脈での再構成）             │
│  - 上長は管理画面で進捗をモニター                       │
│  → 完走 → 試験                                        │
└─────────────────────────────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────┐
│ 試験 + 合否判定（AI推奨 + 人間決裁）                   │
│  - 100問とは別の応用問題                               │

```

このシステムは**真ん中のレイヤーの自動化**に責任を持つ。前後の人間の判断を奪わない。

## 4. AIの4つの役割

このシステムにおけるAIは、4つの異なる仕事をする。それぞれ別のシステムプロンプトを持つ別のエージェントとして設計するのが正しい（混ぜると劣化する）。

### 役割A：出題者（Question Presenter）
- 100問の中から次の問いを選ぶ
- スタッフのこれまでの回答パターンを見て、順序を調整する余地あり（v2）
- MVPでは固定順序でOK

### 役割B：解説者（Explainer）
- スタッフの回答を受けて、KINNOBUTA文脈での再構成を返す
- 「あなたの答えはこう、KINNOBUTA基準だとこう、なぜならこう」という3層構造
- Michaelさんの言葉・比喩・原則を学習している
- 川崎さん版でいう「川崎節で話すエージェント」がここ

### 役割C：採点者（Scorer / Test Examiner）
- 試験フェーズで答案を採点
- ポテンシャル判定（KINNOBUTA思考との適合度）
- 「おすすめ度」を出す
- **合否を出さない**（人間の領域）

### 役割D：観察者（Observer / Coach）
- 上長向け管理画面の裏側で動く
- 各スタッフの回答ログから「詰まりポイント」を抽出
- Michael / 店長に「このスタッフは XX で詰まっている」と報告
- v2機能だが、データモデル設計には今から含めておく

## 5. 多言語アーキテクチャ

### 方針

**コンテンツ（100問の中身）は1セット。AIの「話し方」を言語ごとに分ける。**

### 構造

- 質問本文：英語マスター + フィリピノ語（タガログ）翻訳
- AI解説：英語版エージェント、タガログ版エージェント、それぞれ独立にチューニング
- スタッフはログイン時に言語を選択
- 進捗・ログは言語に依存しない（同じデータベース）

### なぜ翻訳ではダメか

「粛々と」「自己定義の価値」みたいなコア概念は直訳すると死ぬ。
タガログでこれをどう表現するかは、翻訳ではなく**別の哲学エージェント**として育てる必要がある。Michaelさんが日本語ネイティブでタガログのチューニングが難しい場合は、信頼できる現地スタッフを共著者にする運用が必要。

### MVPでの妥協

**MVPは英語版のみ。** タガログ版は構造として準備しておくが、コンテンツはv2で実装。「動くもの」を早く触る方が優先。

```

  - id
  - title

  - examples
  - related_question_ids

Response（回答ログ）
  - id
  - staff_id
  - question_id
  - answer_text
  - ai_explanation（AIが返した解説）
  - time_spent_seconds
  - timestamp

ExamAttempt（試験）
  - id
  - staff_id
  - started_at, completed_at
  - questions_json（出題された問題セット）
  - answers_json
  - ai_score
  - ai_recommendation（AIのおすすめ度: strong_pass / pass / borderline / fail）
  - ai_notes（AIの所見）

  - human_decided_by
  - human_decided_at

ManagerObservation（上長観察ログ・v2）
  - id
  - staff_id
  - manager_id
  - note
  - timestamp
```

## 7. 画面構成

### スタッフ向け

1. **ログイン**（言語選択 + ID）
2. **進捗ダッシュボード**（100問のうち何問終わったか）
3. **問題ページ**（1問表示 → 回答 → AI解説 → 次へ）
4. **試験ページ**（時間制限あり、複数問一括）
5. **試験結果待ちページ**（人間決裁待ち状態の表示）

### 上長（Michael / 店長）向け

1. **スタッフ一覧**（在籍スタッフの進捗が一目でわかる）
2. **個別スタッフ詳細**（回答ログ全部、詰まりポイント、AI所見）
3. **試験決裁ページ**（AIのおすすめ + 人間の合否入力）
4. **質問・原則の編集ページ**（v2、最初はDB直接いじってOK）
5. **詰まりポイント分析**（v2、複数スタッフ横断でどの問いが難所か）

## 8. 技術スタック（推奨）

川崎さん版とほぼ同じ構成にする。理由：Claude Codeに頼みやすい、情報が多い、Vercelで安く動かせる。

| 層 | 技術 | 理由 |
|---|---|---|
| フロントエンド | Next.js (App Router) + TypeScript | Claude Codeが最も得意 |
| スタイル | Tailwind CSS | 高速 |
| 認証 | NextAuth (Google OAuth) | 川崎さん版と同じ。Michael / 店長アカウントの管理が楽 |
| DB | PostgreSQL (Vercel Postgres or Supabase) | 関係性が多いのでRDBMS |
| ORM | Prisma | スキーマ管理が楽、Claude Codeとの相性良い |
| AI | Anthropic Claude API (Sonnet 4) | Michaelさんが既に慣れている |
| ホスティング | Vercel | 川崎さん版と同じ |
| ドメイン | training.kinnobutagroup.com 等 | 後で取得 |
| メール送信 | Resend | 試験合否通知用 |

## 10. MVPスコープ（最初に作るもの）

「動くものを早く触る」を最優先に、以下に絞る：

### 含める
- スタッフログイン（Google OAuth、英語のみ）
- 問題1問〜10問だけ（100問は後で）
- 役割B（解説者）の英語版
- 進捗保存（Response テーブル）

### 含めない（v2以降）
- 試験機能（採点者）
- タガログ版
- ブランド固有モジュール
- 詰まりポイント分析
- 質問編集UI（DB直編集でOK）
- メール通知

### MVP完成の判断基準

> Michaelさんが自分のスタッフに「これ、ちょっとやってみて」と渡せる。

これで十分。完璧を目指すと永遠に出ない。

## 11. 開発フェーズ

### Phase 0: 哲学コアの執筆
- `kinnobuta-os.md`の初稿（Michaelさんが書く、AIドラフト → 添削）
- 100問のうち最初の10問
- これは**コードを書く前に終わらせる**

### Phase 1: MVP実装（Claude Codeで）
- 上記スコープを2〜3週間で実装
- Vercelデプロイ
- Michael本人が10問を通しで体験する

### Phase 2: スタッフ実投入
- 1人の新人で試す（パイロット）
- ログを見て、問いの設計・解説の質を直す
- 100問まで拡張

### Phase 3: 試験機能
- 採点者（役割C）を実装
- 上長決裁フローを実装

### Phase 4: タガログ + 上長分析
- 多言語対応
- 観察者（役割D）と詰まりポイント分析

### Phase 5: ブランド固有モジュール
- HIBI / KINNOBUTA / ZETTON のオペレーション層を追加
- 各店長を巻き込む

## 12. 成功指標（後で測れるように）

MVPでは測らないが、構造としては最初から含めておく：

- スタッフが100問を完走するまでの時間
- 詰まりポイント分布（どの問いで時間が長くなるか）
- 試験合格率
- 試験合格者のOJT初期パフォーマンス（店長の主観評価で十分）
- 「次の日来なくなる」率の変化（Day 0スクリーニング後）

最後の指標が一番重要。**「来続けてくれる人」が増えれば、このシステムは勝っている。**

## 13. リスクと先回り

| リスク | 先回り |
|---|---|
| 哲学ファイルが薄くて、AIが「KINNOBUTAっぽくない」答えを返す | Phase 0で時間をかけて書く。妥協しない。 |
| スタッフが座学を退屈に感じて離脱する | サジェスト機能（川崎さん版にもあった）で対話的にする。一方的な説明にしない。 |
| 試験のAI推奨と人間決裁がズレる | これはむしろ良い兆候。ズレを記録して、AIプロンプトを学習データで更新 |
| タガログ版の品質が出ない | 信頼できる現地スタッフを翻訳・チューニングの共著者にする |
| 100問が「正解暗記ゲーム」になる | 自由記述メインにする。選択肢は最小限。AIの解説は思考プロセスを引き出す問いを返す |
| Michaelさんが忙しくて哲学ファイルが書けない | Claude（このチャット）に対話形式で引き出してもらう。一気に書かない |

## 14. 川崎さん版から学んでいること

| 川崎さんの気づき | このシステムへの反映 |
|---|---|
| 採点マシンだけでは足りない、相談が要る | 100問それぞれに**AI解説**をつける（一問一答ではない） |
| エージェントは本人の言葉で話すべき | `kinnobuta-os.md`を最重要ファイルとして扱う |
| 会話が止まる問題 → サジェスト機能 | 解説後に「もっと深掘りする？」「次に進む？」のサジェスト |
| 一つのアイデアが複数の問題を解く | 教育・スクリーニング・観察を一つのシステムで解く |
| ログが自分を成長させる | 観察者（役割D）として最初から設計に組み込む |
